#标题
[SCTP测试脚本在非交互式环境因stdin重定向导致传输中断]

#内核版本
[2403sp3]

#问题现象
[1. 执行`sctp_darn -s`时触发`cat: -: Bad file descriptor`错误；  
2. SCTP服务端日志显示`recvmsg() returned 0 !!!!`，连接异常终止；  
3. 后续`cat server_output`报错`No such file or directory`，因预期输出文件未生成；  
4. 测试用例`oe_test_lksctp-tools_sctp_status`因文件缺失判定失败；  
5. 问题仅在cloudpods平台非交互式（无TTY）执行时复现。]

#问题根因
[在cloudpods平台以非交互式、无TTY方式执行脚本时，平台将标准输入（stdin）重定向至/dev/null：一方面导致依赖stdin读取数据的cat命令触发文件描述符错误；另一方面使sctp_darn -s因无输入数据无法完成SCTP数据包传输（recvmsg返回0），进而未生成预期的server_output文件。]

#解决方案
[将脚本中引发stdin依赖的`cat`命令替换为`sleep 10`，核心作用是延长进程存活时间，保障SCTP连接有足够时长完成完整的数据传输流程，避免因进程过早退出导致传输失败、server_output文件无法生成的问题。]
